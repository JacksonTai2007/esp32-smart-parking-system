# 答辩演示脚本

目标：5~8 分钟内完整展示"来车 → 刷卡 → 开闸 → 车位更新"闭环，
并展示 Web 远程监控。建议提前一天完整彩排一次并录屏备份。

## 演示前检查清单（提前 30 分钟）

- [ ] ESP32 已烧录最新 main 分支固件
- [ ] 演示用白名单卡 2 张、未登记卡 1 张，已验证 UID
- [ ] SG90 外接电源电量充足，闸杆动作正常
- [ ] OLED 显示正常
- [ ] 手机已能连上 ESP32（现场无 Wi-Fi 时用 AP 模式：
      连热点 `ESP32-Parking`，浏览器开 `http://192.168.4.1/`）
- [ ] 红外车位传感器灵敏度已用演示小车模型调好
- [ ] 备份：演示录屏存在手机里（硬件现场翻车时播放）

## 演示流程

### 场景 1：系统启动（30 秒）
上电 → OLED 显示 Smart Parking 启动画面 → 进入状态页。
讲解：ESP32 主控 + 模块化固件架构（可带一句云端 AI 协作开发流程）。

### 场景 2：车辆入场——合法卡（2 分钟，核心）
1. 小车模型靠近入口 HC-SR04（<20cm）
2. OLED 显示 "Please swipe card"，Web 页入口状态变"等待刷卡"
3. 刷**白名单卡** → 蜂鸣器短响 1 次 → 闸机抬杆 → OLED 显示 Welcome
4. 把小车放进车位 → 红外触发 → OLED 与网页剩余车位 4→3
5. 5 秒后闸机自动落杆

### 场景 3：无效卡拒绝（1 分钟）
车辆再次靠近 → 刷**未登记卡** → 蜂鸣器短响 3 次 → 闸机不动 →
OLED 显示 "Invalid card!"。
讲解：白名单存于固件，串口可见每张卡 UID，添加白名单只改一处配置。

### 场景 4：车位已满（1 分钟）
遮挡全部车位传感器 → 车辆靠近 → 蜂鸣器长响 1 次 → 不提示刷卡、不开闸 →
OLED/网页显示 Parking FULL。移开一个遮挡 → 自动恢复"等待刷卡"。

### 场景 5：Web 远程监控与控制（1.5 分钟）
手机展示仪表盘：实时车位图、闸机状态、最近刷卡 UID（1 秒刷新）。
点"手动开闸/关闸"演示远程控制，讲解 REST JSON API（`/api/status`）。

### 场景 6：收尾（30 秒）
点网页"重置演示"恢复初始状态。总结 Phase 2 规划（烟雾/火焰联动风扇等）。

## 现场故障预案

| 故障 | 预案 |
| --- | --- |
| RFID 不读卡 | 用 Web "手动开闸"完成流程演示，口述刷卡逻辑 |
| 超声波误触发 | 演示前调大/调小 `ULTRASONIC_DETECT_CM` 重新烧录；或直接讲解状态机日志 |
| 现场无 Wi-Fi | AP 模式（固件自动兜底），手机连 ESP32 热点 |
| 舵机无力/抖动 | 检查外接电源；不行就以蜂鸣器+OLED 状态代替闸机动作说明 |
| 全面翻车 | 播放彩排录屏 + 讲串口日志截图 |

## 答辩讲解要点

- 非阻塞 `millis()` 状态机：为什么不用 `delay()`（多外设并发响应）
- 模块化：8 个单一职责模块，引脚/参数集中配置，可维护性
- 工程协作：云端 AI Agent 开发 + CI 自动编译 + 本地硬件验证的完整流程
- 安全细节：Echo 分压、舵机独立供电共地、输入专用脚的使用
