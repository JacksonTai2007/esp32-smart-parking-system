# 答辩演示脚本

目标：5~8 分钟内完整展示"车位识别 → 开始计时 → 离场计费 → 车位/收入更新"闭环，
并展示 Web 远程查看与清零。建议提前一天完整彩排一次并录屏备份。

> 本系统：入口触摸感应识别车辆到达并自动分配车位，红外车位传感器识别
> 进出与计费（无 RFID 刷卡、无舵机闸机）；"收费"仅指本地费用计算与展示，
> 不接任何真实支付。

## 演示前检查清单（提前 30 分钟）

- [ ] ESP32 已烧录最新 main 分支固件
- [ ] 2 路车位红外 + 1 路入口红外灵敏度已用演示小车模型调好，遮挡/移开判定稳定
- [ ] 火灾报警已测：红外遥控器对准火焰传感器一按即报警（警报 + 风扇 + 抬杆），移开约 2 秒自动解除
- [ ] 入口触摸感应一拍即识别到达并分配车位（网页横幅 + OLED `Go to Px`）
- [ ] OLED 显示正常（标题、Free x/总数、车位地图、累计收入）
- [ ] 蜂鸣器声音正常（入场 1 短 / 离场 2 短 / 满位 1 长）
- [ ] 手机已能连上 ESP32（现场无 Wi-Fi 时用 AP 模式：
      连热点 `ESP32-Parking`，浏览器开 `http://192.168.4.1/`）
- [ ] 计费单价/免费时长已按演示节奏调好（`Settings.h`，免费时长别设太长，
      否则现场来不及产生费用）
- [ ] 备份：演示录屏存在手机里（硬件现场翻车时播放）

## 演示流程

### 场景 1：系统启动自检（30 秒）
上电 → 串口（115200）打印 banner `ESP32 Smart Parking System (slots + billing)`、
随后各模块 `init OK`（Alert / OLED / Slots / Bill）→ OLED 显示 Smart Parking 启动画面
后进入状态页（Free x/总数、车位地图、累计收入）。
讲解：ESP32 主控 + 模块化固件架构（可带一句云端 AI 协作开发流程）。

### 场景 2：无感入场——自动放行 + 分配车位引导（2 分钟，核心）
1. 小车驶近入口、遮挡入口红外（模拟模式：点网页「车辆到达入口」按钮）
2. 蜂鸣器**短响 1 次** → 识别车辆到达（无需刷卡取票），系统自动分配编号最小的空闲车位
3. 网页弹出蓝色引导横幅「已分配车位 Px · 请按指引驶入」、目标车位高亮 ⬇「在此停车」；OLED 显示 `Entry: go to Px`
4. 按引导把小车放进被分配车位、遮挡该车位红外 → 引导自动消失、开始计时，`Free x/总数` 减 1，网页该车位变 🚗 实时累计
5. 讲解：入口触摸只产出"来车事实"，分配决策集中在 ParkingManager；分配 30 秒未停入自动作废；满位时到达入口会被拒绝（长响提示）

### 场景 3：车辆离场——按时长计费（2 分钟，核心）
1. 把小车移开、解除该车位遮挡
2. 蜂鸣器**短响 2 次** → 识别为离场，按停留时长结算费用
3. OLED：车位地图对应位变 `_`、`Free x/总数` 加 1、消息显示本次结算
   （如 `P2 left 00:35  0.00` 免费时长内为 0，或 `P2 left 03:12  1.50`）、
   `Total` 累计收入与停车次数 `n` 同步更新
4. Web 页：累计收入/停车次数更新，"最近停车记录"追加一条（车位号·时长·费用）
5. 讲解计费规则：免费时长内为 0；超出后按分钟向上取整 × 单价，金额用整数「分」运算

### 场景 4：车位已满提示（1 分钟）
遮挡全部车位传感器，使最后一个车位由空闲变占用 → 蜂鸣器**长响 1 次** →
OLED 消息显示 `Px in - Parking FULL`、`Free 0/总数`、Web 剩余车位为 0。
移开任一遮挡 → 该车位离场结算、恢复有空位。

### 场景 5：Web 远程查看与清零（1.5 分钟）
手机展示仪表盘（每秒刷新）：总/已占用/剩余车位、车位网格（占用显示实时停留时长）、
累计收入/停车次数/计费单价、最近停车记录列表、系统消息与运行时间。
点"清零累计收入"（确认后 `POST /api/reset`）→ 累计收入与最近记录归零，
在场车辆计时同步重置（相当于本轮演示重新开始）。
讲解 REST JSON API（`GET /api/status`），以及该接口可由 `ENABLE_WEB_MANUAL_CONTROL`
关闭（关闭后返回 403、仪表盘只读）。

### 场景 6：火灾报警与风扇联动（1 分钟）
1. 用红外遥控器对准火焰传感器按任意键（安全触发，无需明火）
2. 蜂鸣器**循环警报** + **风扇自动启动** + 网页红色警报横幅，OLED 消息 `FIRE ALARM - fan ON`
3. 移开触发源约 2 秒 → 警报自动解除、风扇停止、横幅消失
4. 讲解：火焰判定带确认 / 解除时间防误报，警报解除后系统自动恢复正常

### 场景 7：收尾复位（30 秒）
移开全部小车使车位回到全空 → 点网页"清零累计收入"恢复初始统计。
总结 Phase 2 规划（进出场计数、数据持久化、烟雾/火焰报警联动风扇等）。

## 现场故障预案

| 故障 | 预案 |
| --- | --- |
| 车位传感器电平相反（遮挡反而显示空闲） | 改 `config/Settings.h` 的 `SLOT_OCCUPIED_LEVEL`（LOW↔HIGH）重新烧录 |
| 车位状态频繁跳变/误触发进出场 | 调大 `SLOT_DEBOUNCE_MS`，或讲解去抖日志（`[Slots] slot N -> ...`） |
| OLED 不亮 | 确认 I2C 地址 `0x3C/0x3D`（改 `OLED_I2C_ADDRESS`）与 SDA/SCL 接线，串口看 `[OLED] init` |
| 现场来不及产生费用 / 单价不合适 | 改 `Settings.h` 的 `PARKING_FREE_PERIOD_SEC`、`PARKING_RATE_PER_MIN_CENTS` 重新烧录 |
| 现场无 Wi-Fi | AP 模式（固件自动兜底），手机连 ESP32 热点 `ESP32-Parking` |
| 全面翻车 | 播放彩排录屏 + 讲串口日志截图（`[Slots]`/`[Bill]`/`[Gate]` 日志最直观） |

## 答辩讲解要点

- 非阻塞 `millis()` 调度：为什么不用 `delay()`（车位采集/计时/蜂鸣/OLED/Web 并发响应）
- 模块化：单一职责模块（车位 / 业务 / 计费 / 提示 / 消防 / 显示 / Web），引脚与参数集中配置
- 智能入场：触摸感应产出"来车事实"，ParkingManager 集中做分配决策与超时作废
- 计费设计：整数「分」运算避免浮点误差、按分钟向上取整、免费时长抗短暂停靠误触发
- 工程协作：云端 AI Agent 开发 + CI 自动编译 + 本地硬件验证的完整流程
- 安全细节：GPIO 34/35 输入专用脚的使用、外接设备共地、计费"仅本地展示不收款"的边界
